# 0002. モノレポ化せず別リポジトリで運用

- 状態: 採用
- 日付: 2026-04-23

## 背景

EduEvidence JP と EduWatch JP は同一運営者・同一ブランド体系・同一技術スタック(Astro 6 + React 19 + Tailwind 4 + TypeScript)で動く。共通コンポーネントの重複が発生する一方、編集サイクル(ストック型 vs フロー型)と更新頻度(月次 vs 日次)は大きく異なる。

両サイトを同じリポジトリで管理するか、別リポジトリで管理するかの判断が必要だった。

## 検討した選択肢

- **A) モノレポ(同一リポジトリ)で `apps/edu-evidence` と `apps/edu-watch` を並列配置、共通コンポーネントを `packages/shared` に抽出**
- **B) 別リポジトリ。共通コードは必要最小限を **コピー持ち込み** で管理**
- **C) 別リポジトリ。共通コードを npm パッケージ(GitHub Packages)で配布**

## 決定

**B) 別リポジトリ + コピー持ち込み** を採用。

### コピー持ち込み対象(2026-04-23 時点)

- `src/layouts/Layout.astro`
- `src/data/glossary.ts`(用語集)
- `src/styles/global.css`(デザイントークン)
- `src/lib/og-image.ts`
- `src/plugins/remark-glossary.mjs`
- `public/_headers`(セキュリティヘッダー)

### 共通化の判断タイミング

半年運用後(2026-10 頃)に、以下の条件で再評価:

- 共通コードの重複改修コストが大きくなっている
- デザイントークンの差分(色 / フォント / 余白)が頻繁に変動している
- 両サイトで共通の機能要求(例: 同じ検索 UI)が複数発生している

条件が満たされていれば、その時点でモノレポ化または npm パッケージ化を検討する。

## 帰結

### 良い帰結

- リポジトリの context が小さく、運営者・LLM・新規貢献者すべてが理解しやすい
- Cloudflare Pages のビルド設定がシンプル(リポジトリ単位)
- edu-watch 固有の依存(`rss-parser` / `cheerio` 等)を edu-evidence に持ち込まずに済む
- ガバナンス(ブランチ保護 / Dependabot / Issue ラベル)を独立して設計できる

### トレードオフ

- コピー持ち込みファイルの差分管理を運営者が手動で意識する必要がある
- 大規模な共通改修は両リポジトリにそれぞれ PR を立てる手間が発生する
- 共通リファクタリングの恩恵を 2 リポジトリ同時に受けられない

## 撤回 / 再検討の条件

- 共通コードの重複改修コストが想定を超えた場合(半年後の再評価で判定)
- 第三のサイト(SaaS / 法律サイト)が立ち上がり、3 リポジトリ並走の管理が破綻した場合
- 月 1 回以上、両サイトに同じ修正を二度入れている状態が 3 ヶ月続いた場合
