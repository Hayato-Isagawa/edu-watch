# 0005. X 統一運用と Email Routing 3 エイリアス

- 状態: 採用
- 日付: 2026-04-24

## 背景

姉妹サイト 2 つ(edu-evidence と edu-watch)を運営する上で、ソーシャル発信窓口とメール窓口の設計が必要だった。

### ソーシャル

- サイトごとに別アカウントを持つと、運営者の発信工数が倍になる
- 一方、サイトを跨いだ「教育エビデンス x 時事」の横断発信は読者にとって価値が高い

### メール

- お問い合わせ / 報道 / システム通知の窓口を 1 つに統一すると、用途別の優先度判定が難しい
- 一方、複数アドレスを持つと運営者の確認漏れリスクが増える

## 検討した選択肢

### ソーシャル

- **A) サイトごとに独立した X アカウント(`@edu_evidence` と `@edu_watch_jp`)**
- **B) 統一アカウント(`@edu_evidence` のみ)で両サイトを発信**
- **C) ソーシャル発信そのものを当面行わない**

### メール

- **D) 1 アドレス(`info@edu-evidence.org`)に集約**
- **E) 用途別に 3 エイリアスを作り、Cloudflare Email Routing で運営者の Gmail に転送**
- **F) サイトごとにメールアドレスを分ける**

## 決定

### ソーシャル → **B) X 統一運用**

`@edu_evidence` を統一発信窓口とし、edu-watch の更新も同アカウントから流す。週次ダイジェスト発行時に **1 スレッド投稿** で発信する運用(PRD §9.3 に記載)。

### メール → **E) 3 エイリアス + Email Routing**

Cloudflare Email Routing で 3 エイリアスを `eduevidence.jp@gmail.com` に転送:

| エイリアス | 用途 |
|---|---|
| `contact@edu-evidence.org` | 一般お問い合わせ |
| `news@edu-evidence.org` | 報道 / 取材 |
| `notify@edu-evidence.org` | システム通知(Dependabot / Cloudflare 通知 / Bot コミット 等) |

コミットボットは `edu-watch-bot <notify@edu-evidence.org>` を使う。

## 帰結

### 良い帰結

- 発信工数が半分で済み、姉妹サイト構造を読者に直感させやすい
- フォロワー基盤を 1 アカウントに集中でき、初期立ち上げのリーチが取りやすい
- メールは Gmail 1 受信箱に集約され、ラベルで用途を切り分けられる
- システム通知が `notify@` に隔離され、人間からの問い合わせ(`contact@` / `news@`)を見落とさない
- DMARC / SPF / DKIM の設定対象が edu-evidence.org 1 ドメインのみで済む

### トレードオフ

- 投稿者がサイト切り分けを文中で明示する手間がある(「[EduWatch]」プレフィックスなど)
- アカウント停止リスクが 1 つに集中
- 将来サイトが増えた場合、X が手狭になる可能性(その時点で再評価)

## 撤回 / 再検討の条件

- フォロワーが一定規模を超え、サイトごとに独立した発信が編集上必要になった場合
- X のアカウント運用ポリシーが変更され、複数サイト発信が困難になった場合
- 第 3 のサイトが立ち上がり、X 単一アカウント運用が破綻した場合
