# 0043. fetch-news の JST 18:00 cron を JST 19:30 へ微調整(GitHub Actions ピーク帯遅延緩和)

- 状態: 採用(0009 §決定の cron 式を更新)
- 日付: 2026-05-15
- 関連 ADR: 0009(fetch-news ワークフロー PR フロー化、cron 日次 2 回)/ 0042(token 切替、required check 発火条件)
- 関連 PR: TBD(本 ADR 起票 PR)
- 撤回 / 再検討トリガー: 本 ADR §「撤回 / 再検討の条件」参照

## 背景

ADR 0009 で確定した cron スケジュール `0 22,9 * * *`(JST 07:00 / 18:00 = UTC 22:00 / 09:00)について、運用開始後の実測で JST 18:00 cron に構造的な遅延が発生していることが確認された。

直近 8 日(2026-05-07 〜 05-14)の `gh run list -R Hayato-Isagawa/edu-watch --workflow=fetch-news.yml` 計測:

| 日付 | JST 07:00 (UTC 22:00) 遅延 | JST 18:00 (UTC 09:00) 遅延 |
|---|---|---|
| 05-14 | 47 分 | **2h 08m** |
| 05-13 | 54 分 | **2h 24m** |
| 05-12 | 51 分 | **2h 10m** |
| 05-11 | 44 分(failure) | **2h 55m** |
| 05-10 | 35 分 | 1h 04m |
| 05-09 | 31 分 | 59 分 |
| 05-08 | 43 分 | 1h 16m |
| 05-07 | 50 分 | — |
| **平均** | **約 45 分** | **約 1h 55m** |

JST 07:00 cron が 30〜50 分の安定遅延に収まる一方、JST 18:00 cron は平均 1h55m、特に直近 4 日連続で 2 時間超に悪化。

### 原因

- UTC 09:00 = ロンドン 09:00 = ヨーロッパビジネスアワー開始時刻にあたり、GitHub Actions の hosted runner キューがピーク帯
- GitHub 公式ドキュメントは [`schedule` イベントは "best effort"](https://docs.github.com/en/actions/using-workflows/events-that-trigger-workflows#schedule) でピーク時に大幅遅延する旨を明記
- 加えて `0 9 * * *` のような minute=0 ピンポイントは "on-the-hour" 殺到で遅延が累積しやすい(最も混雑する時刻パターン)
- UTC 22:00 cron は欧米深夜帯でピーク外のため遅延が小さい

### 業務上の影響

ADR 0009 で JST 18:00 cron に与えた業務要件は「夕刊感覚、退勤後・帰宅後、行政発表(午前〜午後集中)を同日中に拾う」。実遅延 JST 20:08 〜 20:55 でも教員の退勤後カバーは確保できているが、想定より約 2 時間遅い反映でユーザー体験に劣化が見られる。

## 検討した選択肢

### A. JST 22:00(UTC 13:00)に移動

「夕方読み逃した分を夜にチェック」用途。退勤後カバーは維持され、ピーク帯(UTC 09-12)を抜けて 1 時間程度の改善が見込める。ただしヨーロッパ昼〜夕方帯でピーク傾向は残る。

### B. JST 23:00(UTC 14:00)に移動

「就寝前チェック」用途。ピーク帯から完全離脱し、JST 07:00 cron 並みの 30〜50 分遅延に収まる見込み。ADR 0009 の「夕刊感覚」要件からは離れる。

### C. JST 19:30(UTC 10:30)に微調整(採用)

JST 18:00 を 30 分ずらし、minute=0 殺到から外す最小変更。ピーク帯内のため改善効果は限定的だが、ADR 0009 の業務要件を最大限維持できる。

## 決定

**C(JST 19:30 / UTC 10:30 への微調整)** を採用する。

`.github/workflows/fetch-news.yml` の `schedule:` を以下に更新:

```yaml
schedule:
  # JST 07:00 = UTC 22:00 (-1d)(設計書 §6.1、ADR 0009)
  - cron: "0 22 * * *"
  # JST 19:30 = UTC 10:30(ADR 0043: ピーク帯 + on-the-hour 殺到回避のため JST 18:00 → 19:30 に微調整)
  - cron: "30 10 * * *"
```

JST 07:00 cron は遅延が許容範囲のため変更しない。`workflow_dispatch` も維持。

### 採用理由

- ADR 0009 の業務要件(夕刊感覚、退勤後・帰宅後、当日中の行政発表反映)を最大限維持
- yml 1 ブロックの変更で済み、影響範囲が狭い
- minute=30 へのずらしは on-the-hour 殺到回避効果があり、ピーク帯内でも一定の改善が見込める
- 効果が不十分な場合は次段階で時刻自体を A / B に変更する余地を残す(撤回条件として明記)

## 帰結

### 良い帰結

- ADR 0009 の「夕刊感覚」業務要件を維持したまま遅延緩和を試行できる
- 変更が最小で revert も容易

### トレードオフ

- UTC 10:30 はピーク帯(UTC 09-12)内のため、改善効果が JST 22:00 / 23:00 への移動より小さい可能性
- 効果測定のため最低 1 週間の運用観測が必要

## 撤回 / 再検討の条件

- 本 ADR 採用後 1 週間(7 cron 実行)で平均遅延が JST 07:00 cron(約 45 分)の 2 倍(90 分)を超える場合 → A(JST 22:00)または B(JST 23:00)へ移動を検討
- ADR 0009 の業務要件(夕刊感覚)が読者から不要と判断された場合 → 日次 1 回(JST 07:00 のみ)に削減を検討(ADR 0009 §B4 案)
