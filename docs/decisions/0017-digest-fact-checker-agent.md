# 0017. 週次ダイジェストに事実検証サブエージェント `digest-fact-checker` を導入する

- 状態: 採用
- 日付: 2026-04-30
- 関連 PR: TBD(本 ADR の実装)、初回ダイジェスト PR #50(本フローを使わずに執筆した運用 1 回目)

## 背景

Sprint 4 の初回ダイジェスト(`src/content/digests/2026-04-25.md`)を運用 1 回目として人手レビュー中心で執筆した。事実検証(東京都カスハラ防止条例の年月、千葉市プール水賠償の金額、MIT Media Lab EEG 研究、豊福晋平氏発言、調査主体・サンプリング・主要数字)はすべて執筆代行(Claude Code)が WebSearch / WebFetch で一次資料に当たって裏取りしたが、これは **明示的なエージェント分離による検証ではなく、執筆代行が自己検証** する形だった。

問題は、本フローが定常化すると以下のリスクがある:

- 執筆と検証が同じセッションで進むと、**先に書いた論旨を後から事後正当化** する形になり、検証が甘くなりやすい
- 数値の桁・日付・人物の肩書きなど、見落としやすい要素が **執筆者に依存して網羅性が変動** する
- edu-evidence の `edu-pre-write-verifier` は同様の問題に対し「執筆前ゲート」として独立サブエージェント化することで効果を上げており、edu-watch も同じ流儀に揃える価値がある(ルール 7)

ユーザー指示(2026-04-30)を踏まえ、ダイジェスト執筆前の事実検証を **専用サブエージェント** で構造化する。

## 検討した選択肢

### A. `digest-fact-checker` 専用サブエージェントを `.claude/agents/` に新設(採用)

edu-evidence の `edu-pre-write-verifier` の構造を edu-watch のダイジェスト用語に翻訳して移植。検証観点を 5 つに整理(数値 / 人物 / 日付 / 比較 / 中立性・スコープ)、ツールは検索・閲覧系のみで Edit / Write は持たない(執筆代行をしない)。

メリット: 編集者と執筆代行(別エージェント)とは独立して、事実検証だけに専念するエージェントが入る。執筆代行が自己検証する誘惑を構造的に排除できる。

### B. 既存の `tdd-guide` / `code-reviewer` 系エージェントに事実検証を委ねる(却下)

却下理由: これらはコード品質向け。教育ドメインの事実検証(政府発表・調査・人物発言)のチェックリストを持っていない。

### C. 検証ステップを `docs/sprint-4-design.md` §5 の手順に書くだけで、エージェント分離はしない(却下)

却下理由: 検証手順を文書化するだけでは、執筆代行が手順を「飛ばす」誘惑を防げない。**サブエージェント = ツールの境界** で検証専用の責務を分離した方が、構造的に守れる。

## 決定

A を採用。具体的な実装:

### サブエージェント定義(`.claude/agents/digest-fact-checker.md`)

- name: `digest-fact-checker`
- description: 週次ダイジェストのコメント執筆前ゲート。MUST BE USED before drafting any new digest section comment.
- tools: `Read, Grep, Glob, Bash, WebFetch, WebSearch`(編集系を持たない)
- 必須検証観点 5 項目: 数値・統計 / 人物・肩書き / 日付・出来事 / 比較の妥当性 / 中立性・スコープ
- 出力フォーマット: 総合判定(GO / REVISE / STOP)+ 各観点の判定 + 修正提案 + Sources(一次資料 URL)

### 設計書 §5 の更新

`docs/sprint-4-design.md` §5「編集ワークフロー」を改訂:

- 1 トピックずつ「論旨提示 → `digest-fact-checker` で検証 → 確定 → 執筆代行 → 確認 → 書き込み」のループを回す手順に正式化
- §5.3 を新設し、エージェントの位置付け(配置・ツール・役割・出力)を明記

### 運用上の効果(期待)

| 観点 | 効果 |
|---|---|
| 数値・日付の桁ミス | 執筆前に一次資料で必ず確認するため、訂正リスクが下がる |
| 比較の指標不一致 | 「N 倍」「半分」などの倍率表現で異なる指標を比較していないかチェック |
| 中立性 | 主観過剰・断定過剰・政治性を執筆前に REVISE できる |
| 編集所要時間 | 検証は WebSearch / WebFetch で並列に進むため、30 分以内 KPI を圧迫しない見込み |

## 影響

- 新規ファイル: `.claude/agents/digest-fact-checker.md`(エージェント定義)
- 改訂: `docs/sprint-4-design.md` §5(編集ワークフローに検証ゲートを正式化)
- リポジトリ動作への影響: なし(エージェント定義は Claude Code セッション内で読み込まれるのみ、build / runtime に影響しない)
- 編集ワークフロー: 次回(2026-05-02 公開予定)以降のダイジェストから本フローを定常化

## 撤回 / 再検討の条件

- エージェントの False positive(問題ない記事を STOP / REVISE 判定)が頻発する場合 → 観点を調整
- 編集所要時間が 30 分を大幅に超過するようになる場合 → 検証観点の優先順位を縮める / 並列化を強化
- LLM ベースの自動執筆ツールを将来導入する場合 → 検証観点はそのまま流用、執筆代行を切り替える形に拡張
