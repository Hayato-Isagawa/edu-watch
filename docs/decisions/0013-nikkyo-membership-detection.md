# 0013. 日本教育新聞の会員限定記事を判別し、UI 上で予告する

- 状態: 採用
- 日付: 2026-04-28
- 関連 PR: TBD(本 ADR の実装)

## 背景

Sprint 3 のフロント実装で記事一覧を立ち上げる中、日本教育新聞(`nikkyo`)の記事には会員登録(無料 or 有料)が必要なものが混在することが判明した。読者がタイトルをクリックして媒体サイトに遷移した際に会員ログインを要求されるケースがあり、ADR 0008 の「公式 URL に直接リンクし、編集者の意見は加えない」方針は守りつつも、読者体験として事前予告がほしい。

調査の結果、RSS 側には会員限定を示す要素が存在しない。個別記事 HTML 側でも `paywall` / `member-only` のような専用 class は付いていない。一方で、`<article>` 要素のテキスト内に **「会員登録」と「ログインして」が共起しているか** を見ることで判別が可能(サンプル 10 件で 100% 分離):

| 判別 | 記事 |
|---|---|
| 無料 | post-310058 / post-310050 / post-310045 / post-310038 / post-309792 |
| 会員限定 | post-309806 / post-309689 / post-309804 / post-309434 / post-309769 |

`<article>` テキスト全体の文字数(無料 1300〜1700 / 会員限定 1060〜1130)でも分離できるが、文字数閾値は媒体サイトのテンプレ変更に脆弱で、キーワード共起の方が頑健と判断した。

## 検討した選択肢

### A. 現状維持(却下)

会員壁にぶつかった読者の離脱が予告できない。読者体験の劣化を許容することになる。

### B. 凡例セクションに注記(採用、C と併用)

トップページの「ソースについて」セクションに「日教の一部記事は会員登録が必要、該当記事にバッジを付ける」と一行入れる。実装コストゼロ。ただし B 単独では個別記事レベルで「これが会員限定」と分からない。

### C. parser で HTML を取得して判別フラグを付与(採用、B と併用)

`nikkyo.ts` 内で RSS の各 item の URL を並列フェッチし、HTML の `<article>` 領域に「会員登録」+「ログインして」が共起するかで判別。`RawArticle` / `Article` schema に `requiresMembership?: boolean` を追加し、UI でバッジ表示する。

サンプル実測: 10 件並列で 2.6 秒。判別失敗(タイムアウト・5xx 等)は `undefined` のまま進めて取り込み自体はブロックしない。

### D. 会員限定記事を除外(却下)

情報網羅性が低下する。タイトル・要約・公開日は読者に届けられるため、除外まではしない。

### E. nikkyo ソース停止(却下)

Tier 2 の主要媒体を 1 つ失う。共同通信が日 1 件前後で薄いため、Tier 2 全体の記事量がさらに痩せる。

## 決定

B + C を併用する。

### B: 凡例セクションの注記

`src/components/SourcesIntro.astro` の Tier 2 ブロック末尾に「日教の一部記事は会員登録が必要、該当記事には『会員限定』バッジを付けて事前予告する」と説明を追加。

### C: parser での判別とバッジ

- `RawArticle` / `Article` schema に `requiresMembership?: boolean` を追加(optional、`true` の場合のみセット、`false` / `undefined` は省略)
- `nikkyo.ts` で RSS の各 item URL を `Promise.all` で並列フェッチ
- `<article>` 領域内の「会員登録」+「ログインして」の AND 共起で判別
- 判別失敗(タイムアウト 8s、5xx、ネットワークエラー)時は `undefined` のまま進めて取り込みブロックしない
- `ArticleCard.astro` で `article.requiresMembership === true` のとき小さな「会員限定」バッジを媒体バッジ隣に表示
- バッジ色: `#fbe6c2` 背景 / `#7a4a00` 文字 / `#e9c98a` 罫線(警告色だが派手すぎないアンバー系)

## 影響

- **schema 拡張**: `RawArticle` / `Article` に `requiresMembership?: boolean`。後方互換あり(既存記事は `undefined` のまま、保存済み JSON は変更不要)
- **parser のレイテンシ**: nikkyo は +2〜3 秒(10 件並列フェッチ)。GitHub Actions の cron 実行時間は数十秒なので影響軽微
- **既存記事には適用されない**: ADR 0010(mergeDay は既存 id を上書きしない)と整合。本 ADR 実装後の取り込みから新規記事に効く
- **判別の脆弱性**: 日教側のマークアップ変更で false negative になる可能性。四半期に 1 度 smoke で確認
- **他媒体への波及**: 共同通信や他媒体で同様の問題が出た場合は、`detectMembershipRequired()` を `src/lib/membership.ts` などに切り出して共通化を検討

## 撤回 / 再検討の条件

- 日教が paywall を class マーカー(`paywall`, `member-only` 等)や RSS フィールドで明示するように変更した場合 → そのマーカーを使う方式に切り替え
- 別媒体(kkn / kyodo など)で同じ問題が判明し対処が必要になった場合 → 共通判別器に再構成
- 判別の false negative / false positive が増えた場合 → キーワード AND 条件の見直し、または文字数閾値の併用を検討
